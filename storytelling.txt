✅ YES — Use It On The Landing Page
The landing page is where you convince someone in 10 seconds that VectorShift matters. A scrollytelling section here would be extremely powerful and directly relevant to your data.

Here is why it works for VectorShift specifically:

The core story of VectorShift is:

Climate is changing → Temperatures are rising → Mosquitoes are moving → New regions are now at risk

This is a sequential cause-and-effect story. That is exactly what scrollytelling is designed to tell.

❌ NO — Do Not Use It On Explorer, Country Profiles, or Data Pages
Those pages are tools, not stories. A researcher does not want animations while they are working. An animation on the data explorer would be annoying, not impressive.

Rule: Scrollytelling belongs on the marketing/explanation layer. The product itself should be fast, clean, and professional.

Is This Hard to Build?
No. This is one of the easiest impressive things you can build in Next.js.

You do not need video files. You do not need a video player. The "animation" is just your map and some text changing as the user scrolls.

The Technical Approach — Simple and Free
text

What you need:
→ CSS sticky positioning (built into every browser, no library)
→ Intersection Observer API (built into every browser, no library)
→ Framer Motion (already in your stack)
→ Your existing map component (already built)

What you do NOT need:
→ Video files
→ Video hosting
→ A special library for scrollytelling
→ Complex state management
How It Works Technically
text

1. The section has a fixed height (e.g., 500vh — five screens tall)

2. The map container is position: sticky — it stays on screen 
   while the user scrolls through the tall section

3. Intersection Observer watches five invisible "trigger" divs 
   — one for each chapter

4. When a trigger enters the viewport, it fires an event 
   that updates which chapter is active

5. The active chapter number controls:
   → Which text block is visible (fade transition)
   → Which year the map shows
   → Which overlay is active
   → Whether country pulses are playing

6. When the user scrolls past all five chapters, 
   the sticky lock releases and normal page scroll resumes
Approximate Code Structure
text

ScrollytellingSection/
├── ScrollytellingSection.tsx   ← Main container (sticky logic)
├── ChapterText.tsx             ← Left side text blocks
├── ChapterMap.tsx              ← Right side mini map (simplified version)
├── chapters.ts                 ← Data: text, year, map state per chapter
└── useScrollChapter.ts         ← Hook: tracks which chapter is active
This is maybe 3-4 days of focused work after your map is already built. Not weeks. Not a separate project.

Video vs. No Video
You mentioned videos. Let me address this directly.

Option A: Real Video (Not Recommended for MVP)
Some sites like Apple use actual pre-rendered video files where the video plays in sync with scroll position.

Why you should NOT do this:

You need to create the video (screen recording, editing, exporting)
Video files are large (50-200MB) — terrible for performance
Videos need hosting (costs money or uses large storage quota)
On slow connections the video buffers and breaks the effect
Hard to keep in sync with scroll position precisely
The free Supabase storage limit is 1GB — video would eat most of it
Option B: CSS + JavaScript Animation (Recommended)
This is what VectorShift should do. Everything "animated" is actually:

Your existing Mapbox map changing its displayed year
CSS opacity transitions between text blocks
Framer Motion fade animations
A few SVG pulse animations for outbreak markers
Why this is better:

Zero extra file size
Works on any connection speed
Perfectly smooth on any device
Easier to build and maintain
More interactive than video (user controls the scroll pace)
Looks just as impressive as video when done well
Examples of sites that do this without video:

The Pudding — all their animations are D3 and CSS
Our World in Data — chart transitions via JavaScript
Flourish data stories — all animated with canvas/SVG
What NOT To Do
Since you asked about this, let me be very direct about the common mistakes:

Mistake	Why It Hurts You
Putting scrollytelling on every page	Makes the site feel slow and gimmicky
Using it to hide thin content	If your data story is weak, animation makes it worse
Animations that play on loop without user interaction	Distracting and unprofessional
Scroll-jacking (overriding native scroll)	Feels wrong, users hate it, causes motion sickness
Long animations with no skip option	Frustrates returning users and researchers
Making users wait for animation before seeing content	Kills engagement
Video files over 5MB	Destroys performance scores
